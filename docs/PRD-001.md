# orchestra — v1.0.0 PRD

> A Claude Code plugin that gives one developer a multi-agent SDLC pipeline behind a single entry point. v1.0.0 is the first cut: smallest core that preserves the **generator/evaluator separation** thesis and the **document-driven gate** discipline, with a single output root, capability-first default models, and structural drift detection across artifacts.

---

## 1. How to Use This Document <a id="S-USAGE-001"></a>

Every requirement is tagged:

- **P0** — load-bearing for v1.0.0. Without it the plugin's thesis collapses or the install is broken.
- **P1** — ships in v1.0.0 because it earned its spot in the source PRD's ablation.
- **P2** — deferred to v1.1+. Listed for visibility, not for scope.
- **P3** — refused. Tried, measured, dropped — do not re-add without new evidence.

The contract: **build P0+P1, defer P2, refuse P3.** Anything not on this list earns its spot the way the source's v2.6.0 ablation did — measure first, ship second.

---

## 2. Vision <a id="S-VISION-001"></a>

A Claude Code plugin that gives a **single developer** a production-quality multi-agent SDLC pipeline, with **one entry point** (`/orchestra`), zero-setup install, and capability-first default models.

**One-line thesis** — *"The model IS the agent. Build harnesses, not prompt chains."* Agent prompts encode roles and boundaries; the harness (hooks, MCP probes, sprint contracts, drift detection) encodes everything the model cannot reason about reliably alone.

## 4. Problem Statement <a id="S-PROBLEM-001"></a>

Solo developers and small teams using Claude Code want the rigor of a full SDLC — spec, design, code, review, release notes — without:

1. **Building the orchestration themselves.** Every solo developer reinvents "PM → architect → coder → reviewer." The cost is rediscovery; the result is fragile.
2. **Trusting one agent to grade itself.** Self-evaluation blindness is structural, not promptable.
3. **Drifting silently across documents.** PRD says one thing, SDD says another, code does a third. No one notices until the bug ships.
4. **Bringing their own conventions.** Frontmatter contracts, artifact schemas, gate definitions, registration mechanics — inherited from the plugin, not invented per-project.

## 5. Target User <a id="S-PERSONA-001"></a>

**Primary persona** — Senior engineer working alone on professional code, comfortable with Claude Code as a CLI, willing to pay opus-4.7 prices for capability, who wants the structure of a team without hiring one. Java/Spring or TypeScript stack typical; brownfield refactors and feature work in roughly equal measure.

**Secondary** — Plugin contributors and ablation-campaign runners.

**Out of scope** — First-time programmers (the pipeline expects the user to read artifacts), enterprises with ops teams (no multi-tenant, no audit trail beyond local files), cost-constrained users (override defaults to sonnet/haiku).

## 6. Goals <a id="S-GOALS-001"></a>

- **G1 — One entry point.** Every task starts with `/orchestra <natural language>`. No setup wizard.
- **G2 — Generator/evaluator structural separation.** Builders cannot grade their own work. Builders have no `Bash`. `@evaluator` owns verdict.
- **G3 — Document-driven artifacts with drift detection.** Every gate stage produces a Markdown artifact in `<project>/.claude/.orchestra/`; cross-references include content hashes; stale references are detected, not silent.
- **G4 — Zero-setup install.** `claude plugin install hunghlh98/orchestra` is the only step.
- **G5 — Removable by design.** Every component disabled by editing one file or setting one env var.
- **G6 — Configurable model tier.** Capability-first default (opus-4.7-1m); per-agent override in `local.yaml`.
- **G7 — Java + TypeScript-first, language-extensible.** Out-of-box rules for Java/TS; rule system supports adding Go/Python/Kotlin without code changes.

Each goal is *measurable* at v1.0.0 ship time via a named validator. CI runs all checks on every push.

## 8. Core Concepts (Load-Bearing Invariants) <a id="S-INVARIANTS-001"></a>

### 8.1 Generator/Evaluator Separation [P0]

- **Builder agents** (`@backend`, `@frontend`, `@test`) write code/tests but **cannot** assert pass.
- **Builders have no `Bash` tool.** Cannot run tests, start servers, exec scripts. Forces every verdict-bearing run to spawn `@evaluator`.
- **`@evaluator`** is **strict read-only** — no `Write`/`Edit`/`MultiEdit`. Runs tests + runtime probes. Writes only the verdict artifact.
- A builder seeing a green test run is **not** allowed to interpret it as success — only `@evaluator`'s verdict counts.

### 8.2 Sprint Contract Gate [P0]

- Before any code, `CONTRACT-<id>.md` is **co-signed** by `@lead` (drafts) and `@evaluator` (validates + signs).
- Contains: weighted criteria summing to 100, runnable probe specs (`http_probe`, `db_state`, shell), critical-failure conditions, non-goals.
- `signed: false` blocks all `IMPL-*` artifacts.
- One revision round permitted; round-2 trips circuit breaker → human oracle.
- **Skip allowed** for docs-only routes, debug-only routes (no code = no contract).

### 8.3 Document-Driven Pipeline [P0]

- All output roots at **`<project>/.claude/.orchestra/`**.
- Per-feature artifacts in `pipeline/<id>/` with shared id (PRD-001, FRS-001, TDD-001 in folder `001/`).
- Project-singleton artifacts: `architecture/SAD.md`, `runbooks/RUNBOOK-vX.Y.Z.md`, `releases/RELEASE-vX.Y.Z.md`.
- Every artifact has YAML frontmatter; every cross-reference includes both artifact id AND content hash (§8.13).
- Artifacts are **gate** (block until human approves) or **informational** (traceability).

### 8.4 Single Entry Point [P0]

- Exactly one user-facing slash command: `/orchestra`.
- Subcommands: `/orchestra sprint`, `/orchestra release`, `/orchestra commit`, `/orchestra help`.
- Bare `/orchestra <natural language>` is primary; `@product` + `@lead` classify intent.

### 8.5 Three-Tier Tool Discipline [P0]

| Tier | Allowed | Forbidden | v1.0.0 Agents |
|---|---|---|---|
| **Strict read-only** | `Read`, `Grep`, `Glob`, `Bash` (scans/tests only) | `Write`, `Edit`, `MultiEdit` | `@evaluator`, `@reviewer` |
| **Implementation-restricted** | + `Write` (artifacts only) | `Edit`, `MultiEdit`, `Bash` | `@product`, `@lead`, `@ship` |
| **Implementer (no Bash)** | full write toolset | `Bash` | `@backend`, `@frontend`, `@test` |

CI enforces tier placement. New agents that don't fit are wrong, not the tier system.

### 8.6 Three-File Registration [P0]

Every component touches exactly three files:

1. **`.claude-plugin/plugin.json`** — agents listed as **explicit file paths**.
2. **`manifests/install-modules.json`** — components grouped by module with `kind`, `defaultEnabled`, `stability`.
3. **`manifests/runtime-toggles.json`** — env-var name + default + description for each runtime-disable-able component.

Adding a component without all three is a CI failure.

### 8.7 Removability [P0]

Every feature removable by **one** of:

- Agent → remove from `plugin.json.agents`, OR omit from `local.yaml.agents` (per-project disable).
- Hook → remove from `hooks/hooks.json`, OR set `ORCHESTRA_HOOK_<NAME>=off`.
- Skill → remove from `manifests/install-modules.json`, OR set `ORCHESTRA_SKILL_<NAME>=off`.
- Rule → edit `paths:` glob in the rule's frontmatter to exclude target files.
- Command → remove from `plugin.json.commands`.
- MCP → remove from `.claude-plugin/.mcp.json`, OR set `ORCHESTRA_MCP_<NAME>=off`.
- Model override → edit `local.yaml`.

Env-var toggles cover **skill / hook / MCP** only — components without a natural register-and-disable mechanism. Agents, rules, and commands disable via the same key that registers them (`plugin.json.agents`, `paths:` glob, `plugin.json.commands`), so a parallel env-var registry would add no capability.

Multi-file edits to disable a component = design failure.

### 8.8 Skill Token Discipline [P1]

- Every `SKILL.md` ≤ 400 lines. Enforced by `validate.js`.
- Depth lives in `references/` and is loaded on-demand.
- Even with 1M context default, this is hygiene — readable artifacts, smaller diffs, faster grep.

### 8.9 External Dependency Decision Rule [P0]

- **Skills embed; tools expose.**
- Methodology / prompts / checklists → embed inline as `skills/<name>/SKILL.md`.
- Executables / CLIs / servers → wrap as MCP under `.claude-plugin/.mcp.json` with stdlib-only stdio servers in `scripts/mcp-servers/`.
- Hook scripts under `hooks/scripts/` follow the same stdlib-only rule. YAML frontmatter parsing in hooks is hand-rolled per the §8.13 frozen grammar; no external YAML parser dependency.
- Never depend on another Claude Code plugin.

### 8.10 Model Configuration [P0]

**Default for all 8 agents: `claude-opus-4-7` (1M context).**

Per-agent override mechanism in `<project>/.claude/.orchestra/local.yaml`:

```yaml
default_model: claude-opus-4-7
default_context_mode: 1m
agents:
  ship:
    model: claude-haiku-4-5    # downgrade for templated work
    context_mode: default
  backend:
    model: claude-sonnet-4-6   # downgrade for cost-sensitive run
    context_mode: default
```

Resolution at TeamCreate time:
1. Agent default = `default_model` from `local.yaml` (else hard-coded `claude-opus-4-7` with `context_mode: 1m`).
2. Per-agent override in `agents.<name>.model` wins; `agents.<name>.context_mode` overrides the context mode independently.
3. CI validates every override resolves to a known Anthropic model id (per `manifests/known-models.json`) and that `context_mode` is in that model's `supportsContextMode` set.

**Inversion of source PRD §6.10.** Source was cost-first ("opus only when justified"). v1.0.0 is capability-first ("opus default, downgrade when budget warrants"). Documented as deliberate; surfaced in install-time README so users know what they're paying for.

### 8.11 Confidence-Based UX [P1]

- HIGH confidence → execute immediately, zero questions.
- MEDIUM → 1 targeted question via `AskUserQuestion`.
- LOW → 2–3 questions, single round.
- **Never more than 3 questions per request.**
- Circuit breaker: 3 rejection rounds in any review stage → write `DEADLOCK-<id>.md`, escalate.

#### 8.11.1 Confidence inputs

`@lead` produces the HIGH/MEDIUM/LOW classification deterministically from the following signals. Inputs are computed at intent-classification time, after `project-discovery` runs but before any artifact is written.

| Signal | Source | Bias |
|---|---|---|
| `intent_word_count` | user prompt | ≤15 words → +HIGH; ≥40 words → +LOW |
| `project_mode` | `local.yaml` | greenfield → +HIGH; brownfield → 0 baseline; brownfield with `inferred:` SAD sections on dependency path → +LOW |
| `files_touched_estimate` | `project-discovery` glob+grep | <5 → +HIGH; 5–20 → +MEDIUM; >20 → +LOW |
| `sad_section_inferred_count` | `architecture/SAD.md` frontmatter along impact path | every `inferred:` along the path bumps tier down by one |
| `intent_classification_match_score` | §9.5 routing taxonomy lookup | exact-match intent (e.g. "docs", "refactor") → +HIGH; novel intent → +LOW |

The classifier sums signals; tier breakpoints are calibration constants in `skills/task-breakdown/references/confidence-calibration.md` (P0 to ship). Override path: user may set `--confidence high|medium|low` on `/orchestra` to bypass classification (logged as `confidence.user-override` event).

**Worked examples:**

```
GIVEN intent="add a dark mode toggle" AND project_mode=greenfield AND files_touched_estimate=3
WHEN @lead classifies confidence
THEN tier=HIGH (Pattern A — linear handoff, no spec dialogue)

GIVEN intent="rewrite auth to use OIDC across all services"
  AND project_mode=brownfield AND sad_section_inferred_count=2 AND files_touched_estimate=27
WHEN @lead classifies confidence
THEN tier=LOW (Pattern C — wave team)

GIVEN intent="fix typo in README" AND files_touched_estimate=1
WHEN @lead classifies confidence
THEN tier=HIGH (routes via §9.5 docs intent; @product skipped)
```

§9.4 dialogue patterns are selected by this classification.

### 8.12 AgentTeams vs SubAgent [P0]

**Agents are team members; sub-tasks are tools.**

| Primitive | Used for | Lifetime |
|---|---|---|
| **TeamCreate** (AgentTeam) | The 8 orchestra agents during a `/orchestra` run | Run lifetime; explicit teardown |
| **Task** (SubAgent) | Bounded sub-delegation inside an agent's work (focused code lookup, single research call) | Single invocation; result returns |

Implications:
- A `/orchestra <intent>` invocation MUST use `TeamCreate` to instantiate the agent roster.
- Task-spawned sub-agents are NOT orchestra agents and MUST NOT produce gate artifacts.
- `metrics-collector` distinguishes `team.member.invoked` events from `task.subagent.invoked` events.

### 8.13 Document Alignment & Drift Detection [P0]

**Every artifact carries section ids + content hashes; every cross-reference carries `hash-at-write`. Pre-write hook stamps hashes on save; validation script detects drift across artifacts; `confirmed:` drift fails the gate, `inferred:` drift warns.**

#### Section demarcation — heading anchors

Top-level (`##`) headings in artifact bodies carry an HTML anchor immediately after the heading text:

```markdown
## §1 System Context <a id="S-CONTEXT-001"></a>

Section content. Sub-headings and prose belong to S-CONTEXT-001 until the next anchor.

### A nested heading

Still part of S-CONTEXT-001.

## §2 Container Decomposition <a id="S-CONTAINER-001"></a>

Section content for S-CONTAINER-001.
```

Section id pattern: `S-<TYPE>-<NNN>` where `<TYPE>` is an upper-case label and `<NNN>` is a three-digit zero-padded counter unique within the artifact. Anchors render natively as link targets in GitHub view, IDE previews, git diffs, AND the validation script — one syntax, all readers.

#### Hash computation

- A section's content = bytes from its `<a id="...">`-bearing heading line (exclusive) to the next anchor-bearing heading line (exclusive) or EOF.
- Hash = SHA256 of the section's UTF-8 content after normalization: LF line endings, trailing whitespace stripped, the anchor heading line itself excluded.
- Frontmatter is NOT part of any section's hash. Frontmatter has its own `revision:` counter for non-content metadata changes.

#### Frontmatter shape

```yaml
sections:
  S-CONTEXT-001:
    hash: "sha256:abc123..."
    confirmed: true                      # or `inferred: true`
  S-CONTAINER-001:
    hash: "sha256:def456..."
    inferred: true                       # brownfield-derived; pending review
references:
  - type: sad
    id: ""                               # SAD is a singleton; id empty
    section: S-CONTAINER-001
    hash-at-write: "sha256:def456..."    # filled by hash-stamper
  - type: frs
    id: "001"
    section: S-API-001
    hash-at-write: "TBD"                 # hash-stamper resolves on next save
```

#### Frozen frontmatter grammar [P0]

Hand-rolled parsers in `hash-stamper` and `validate-drift.js` rely on a constrained YAML subset. Anything outside this grammar fails as `frontmatter-grammar-violation`:

- **Block style only.** Flow style (`{ }`, `[ ]`) rejected.
- **No anchors or aliases** (`&`, `*`).
- **No multi-document streams.** Additional `---` markers inside the body are body content, not document separators.
- **No tags** (`!!str`, `!!int`, etc.).
- **Booleans must be lowercase** `true` / `false` only — no `True`, `yes`, `on`.
- **Hash values must be quoted strings** — `hash: "sha256:abc..."`, never `hash: sha256:abc...`.
- **Keys** match `[a-zA-Z][a-zA-Z0-9_-]*`.

**Scope:** this grammar applies to artifacts under `<project>/.claude/.orchestra/` parsed by hook scripts and `validate-drift.js`. Plugin-repo documentation (this PRD, `DESIGN-*.md`, etc.) is not bound by it. The grammar is small enough to implement in ~150 lines of stdlib-only Node and large enough to express every example in this PRD.

#### Pre-write hook — `hash-stamper`

Fires on `PreToolUse: Write|Edit|MultiEdit` for paths matching `<project>/.claude/.orchestra/**/*.md` and `**/*.openapi.yaml`. Behavior:

1. Read tool input from stdin (file path + content).
2. Detect heading anchors in body; compute each section's hash.
3. Update `sections.<id>.hash` in frontmatter; preserve `confirmed:`/`inferred:` flags.
4. For each `references[].hash-at-write` that is `TBD` or missing:
   - Resolve the upstream artifact path from `(type, id)` per the deterministic table:
     - `type: sad` → `architecture/SAD.md` (id empty/ignored — singleton)
     - `type: prd|frs|tdd|contract|test|code-review|doc|impl-be|impl-fe|code-design-be|code-design-fe|plan` → `pipeline/<id>/<TYPE>-<id>.md`
     - `type: api` → `pipeline/<id>/API-<id>.openapi.yaml`
     - `type: runbook` → `runbooks/RUNBOOK-<id>.md`
     - `type: release` → `releases/RELEASE-<id>.md`
   - Look up the upstream section's current `hash` from upstream frontmatter.
   - Fill in `hash-at-write`.
5. **Existing `hash-at-write: sha256:...` values are NOT modified.** Re-acknowledgment is explicit: user resets the field to `TBD` before save, hook fills with current upstream hash.
6. If an upstream artifact does not exist OR the referenced section id is not present in upstream:
   - Leave `hash-at-write: TBD-UNRESOLVED`.
   - Append event to `metrics/events.jsonl`.
7. Output the modified payload to stdout; exit 0; write proceeds.

**Latency expectation.** Expected fast: parse + SHA256 + frontmatter merge over ~1KB of YAML. **Not a hard SLA.** Pathological inputs (>500 anchors, >1MB body) are out of scope for v1.0.0 and emit a warning event `hook.hash-stamper.slow` if elapsed >500ms. For a typical feature's PRD→FRS→TDD→CONTRACT chain (~6 hook invocations), end-to-end overhead is well under 1s.

Path resolution is deterministic from `type` + `id`. Validation rejects artifacts living outside the convention path.

#### Validation script — `validate-drift.js`

Walks every artifact under `<project>/.claude/.orchestra/`:

1. Parse frontmatter + body.
2. Recompute current hash for each declared section. If frontmatter `sections.<id>.hash` ≠ recomputed → emit `frontmatter-out-of-sync` (means `hash-stamper` did not run; manual edit happened or hook is disabled).
3. For each `references[].hash-at-write`, look up the upstream artifact's section.
4. If upstream's current hash ≠ `hash-at-write`:
   - Inspect upstream `sections.<id>` flag.
   - `confirmed: true` → DRIFT-REPORT entry with `severity: fail`.
   - `inferred: true` → DRIFT-REPORT entry with `severity: warn`.
5. If `hash-at-write` is `TBD-UNRESOLVED`:
   - During in-flight feature work (folder not yet sealed by `RELEASE-*` reference): `severity: warn`.
   - At release-time (folder sealed): `severity: fail`.
6. Emit `<project>/.claude/.orchestra/DRIFT-REPORT.md` with all findings, grouped by downstream artifact.

Release gate (§11.2) blocks on any `severity: fail`. `severity: warn` is informational.

#### Inferred → Confirmed flip

The `inferred: true` flag is the brownfield grace period. A user moves a section to `confirmed: true` by editing the artifact and changing the flag. The act of writing is the act of confirmation — `hash-stamper` recomputes on save, capturing the confirmed state in the new hash. From that point, drift in this section fails the gate.

v1.0.0 has no dedicated command for the flip; manual edit is sufficient. v1.1+ may add `/orchestra confirm-section <artifact>:<section-id>` if the manual workflow proves friction-heavy.

**Discovering inferred sections.** On brownfield bootstrap, every SAD section starts `inferred: true`. To prevent the user facing an unprioritized wall, `validate-drift.js` emits an "Inferred sections needing review" summary as the **first block** of `DRIFT-REPORT.md`, sorted by reference count (most-cited first). The user works the list top-down; each manual confirm-edit shrinks the list. No command in v1.0.0 — friction is observed, then v1.1 decides whether to automate.

#### Re-acknowledgment workflow

When upstream drifts:

1. `validate-drift.js` flags downstream as drifted.
2. User opens downstream artifact, reviews whether it still holds given the upstream change.
3. **If it still holds:** edit `references.<upstream>[i].hash-at-write` to `TBD`, save. `hash-stamper` resolves to current upstream hash. Drift cleared. Git commit shows the hash change as the audit trail.
4. **If revisions are needed:** revise downstream content, also reset relevant `hash-at-write` to `TBD`. On save, `hash-stamper` recomputes both section hashes and the references.

The git log of the downstream artifact is the audit trail of acknowledgments. Reviewers see the hash change in the diff and ask the right question: "did the author actually re-read upstream?"

#### Hash-stamper failure modes

The hook is P0 and runs on every artifact write. Failure semantics determine whether v1.0.0 can recover from a bad write without losing user content.

| Failure | Behavior | Event |
|---|---|---|
| Markdown parse fails (malformed anchor, truncated section) | Exit 0 (passthrough); leave existing frontmatter `sections.<id>.hash` stale. `validate-drift.js` later catches as `frontmatter-out-of-sync`. | `hook.hash-stamper.parse-failed` |
| Upstream artifact YAML corrupt or missing referenced section | Leave `references[i].hash-at-write: TBD-UNRESOLVED`. Exit 0. `validate-drift.js` warns in-flight, fails at release. | `hook.hash-stamper.upstream-unresolved` |
| Hook script itself crashes mid-rewrite | Wrap body in `try/catch`. On crash, write proceeds with the **original** payload (Claude Code default for unhandled hook errors). User content is never silently mutated by a half-completed stamp. | `hook.hash-stamper.crashed` |
| Concurrent writes from two agents to the same file | **Out of scope for v1.0.0.** Single-writer assumption: only one agent edits a given artifact at a time, enforced by sequential agent execution within a `/orchestra` run. v1.1+ revisits when Pattern C wave teams parallelize. | n/a |

Invariant: hash-stamper **never** blocks a write (`exit 2`). Blocking is the job of `pre-write-check` (secrets) — hash-stamper is observability + auto-fill, not enforcement.

| Conflict | Winner | Rationale |
|---|---|---|
| Velocity vs verdict | `@evaluator` | Always |
| Release vs stability | `@ship` | Ultimate trump |
| WHAT vs HOW | `@product` (WHAT) / `@lead` (HOW) | Domain authority |
| `confirmed:` drift in upstream | Block downstream until re-acknowledged | §8.13 |

**Circuit breaker** — 3-strike rule per stage: write `DEADLOCK-<id>.md`, halt, escalate.

#### 9.6.1 Resuming from DEADLOCK

The circuit breaker is one-way until the human acts. Recovery flow:

1. Human reads `DEADLOCK-<id>.md`, decides next direction, edits the artifact's frontmatter to add `resolution: <abandoned|resolved-with-direction|requires-rescope>` plus a free-text `direction:` field if continuing.
2. User re-invokes `/orchestra <intent>` referencing the feature id (e.g. `/orchestra continue 003` or simply re-stating the original intent — `@lead` recognizes the existing `pipeline/003/` and reads the deadlock).
3. `@lead` reads `resolution:`, decides the next agent in the sequence, and the pipeline continues. `abandoned` closes the feature folder; `requires-rescope` returns to `@product` for PRD revision; `resolved-with-direction` resumes from the stage that struck out.
4. `metrics-collector` logs `deadlock.resolved` with the resolution kind.

v1.0.0 ships **no** dedicated `/orchestra resume` command. The re-invocation path keeps the user-facing surface minimal; v1.1 reconsiders if friction is measured.

```
3. @product + @lead negotiate (Pattern B, one round):
     - @product proposes mode based on intent ("new app" vs "fix bug X")
     - @lead validates against discovered state
     - mismatch → AskUserQuestion (no artifact written)
4. Decision lands in local.yaml:
     project_mode: greenfield | brownfield
     bootstrapped_at: <ISO>
     bootstrapped_by: { product: <agent>, lead: <agent> }
   After write, @lead appends `local.bootstrapped` event to
   metrics/events.jsonl with { project_mode, primary_language, framework }
   payload (see §9.9). This is the first metrics event in a fresh project.
5. SAD initialization:
     - greenfield: stub SAD with empty Context/Container scaffolds
     - brownfield: @lead drafts SAD from observation; every section
       starts with `inferred: true`. User confirms section-by-section
       before downstream features can mark `sad-touched: true`.
```

### 9.12 Per-Project State File [P0]

`<project>/.claude/.orchestra/local.yaml`. **Pure YAML.** Never committed.

```yaml
version: 1
project_mode: greenfield                 # or brownfield
bootstrapped_at: "2026-04-28T..."
bootstrapped_by:
  product: "@product"
  lead: "@lead"
default_model: claude-opus-4-7
default_context_mode: 1m

  ship:
    model: claude-haiku-4-5              # example override
    context_mode: default
sad:
  required: true

  metrics_collector: { enabled: true }
  pre_write_check: { enabled: true }
```

- Hooks merge-preserve other features' blocks on write.
- Add `.claude/.orchestra/{local.yaml,metrics,sessions}` to `.gitignore`.

#### 9.12.1 Configuration precedence

Three configuration surfaces interact: shipped manifests, per-project YAML, runtime env vars. Precedence (highest wins):

```
runtime env var (ORCHESTRA_*=off|on)
  > local.yaml features.<name>.enabled / agents.<name>.model
  > manifests/runtime-toggles.json default
  > manifests/install-modules.json defaultEnabled
```

`validate.js` warns when `local.yaml` enables a component whose `manifests/install-modules.json` entry has `defaultEnabled: false` AND `stability: experimental` — the user is overriding both an off-by-default and an unstable component, which deserves a visible footprint at install/update time.

Conflict resolution at `/orchestra` invocation is read-once: `local.yaml` is parsed at TeamCreate; mid-run edits do not hot-reload. To pick up `local.yaml` changes, end the run and re-invoke.

### 10.6 Inline revision history (D-26 concretization)

## 11. Quality Gates <a id="S-QUALITY-001"></a>

### 11.1 Validation Scripts [P0]

| Script | What it validates |
|---|---|
| `validate.js` | Plugin structure, frontmatter, version symmetry, CHANGELOG↔VERSION gate, rule frontmatter, hook test-wiring, skill reference audit |
| `test-hooks.js` | Contract assertions across hook scripts (exit codes, stdout passthrough, side-effect sandboxing). Hermetic via `HOME` override. |
| `test-agents.js` | Frontmatter shape, tools shape, three-tier discipline, `<example>` block presence, **`model` field is a known Anthropic id** |

| `validate-drift.js` | **New in v1.0.0.** Walks artifact tree; for every `references[].hash-at-write`, compares to current section hash; emits `DRIFT-REPORT.md`. |
| `test-removability.js` | **New in v1.0.0.** For each component in `manifests/install-modules.json`, asserts an env-var entry exists in `manifests/runtime-toggles.json`. Backs G5 (Removability — §6.1). |

All six run on every push and PR.

### 11.2 Release Gate [P0]

| Check | Severity | Enforces |
|---|---|---|
| `SKILL.md` ≤ 400 lines | Fail | Progressive disclosure cap |
| CHANGELOG ↔ VERSION | Fail | Bumping VERSION without a `## [X.Y.Z]` entry fails |
| Builder Bash strip | Fail | No implementer-tier agent has `Bash` |
| `.gitignore` covers `.claude/.orchestra/{local.yaml, metrics, sessions}` | Fail | Prevents commit of per-project state, telemetry, session data |
| `confirmed:` drift in `DRIFT-REPORT.md` | Fail | Stale references to confirmed sections block release |
| `inferred:` drift in `DRIFT-REPORT.md` | Warn | Brownfield inferences are expected to evolve |
| ROADMAP ↔ VERSION | Warn | ROADMAP must mention current VERSION or "shipped" |

**Process invariant** — every release diffs against `ROADMAP.md` in the same commit. No two minor releases within 24 hours without an explicit `RELEASE-CUT.md` artifact.

- **Repo-dev only**: `scripts/*.js` at root (`validate`, `bump-version`, `test-*`, `validate-drift`, `generate-inventory`, `postinstall`, `dashboard`).

`.gitignore` enforces with allow-list:

### 15.1 Document Alignment & Drift Detection — resolved 2026-04-28

§8.13 finalized via brainstorm. All three sub-decisions:

| Sub-decision | Choice | Why |
|---|---|---|
| (a) Section demarcation | **Heading anchors** — `## Title <a id="S-..."></a>` | Renders natively in GitHub/IDE/git diff/validation script. HTML comments lose to comment-stripping renderers; whole-file hashing loses granularity. |
| (b) Hash computation responsibility | **Pre-write hook (`hash-stamper`)** | Stamps section hashes on every save; auto-resolves `references[].hash-at-write: TBD` from upstream. Makes acknowledgment a side effect of normal edits. |
| (c) Stale reference policy | **`confirmed: true` → fail; `inferred: true` → warn.** Re-acknowledgment by resetting `hash-at-write: TBD` and saving. | Brownfield gets a grace period via `inferred:`; flipping to `confirmed:` tightens the gate. The act of writing IS acknowledgment; git log is the audit trail. |

Implementation may now proceed per §8.13 mechanism + §9.9 `hash-stamper` hook + §11.1 `validate-drift.js` script. No separate `DESIGN-DRIFT-DETECTION.md` needed; PRD §8.13 is the authoritative spec.

## 19. Appendix — Worked Example: `/orchestra add password reset to login` <a id="S-EXAMPLE-001"></a>

A canonical end-to-end run. Greenfield TypeScript project, `local.yaml` pre-bootstrapped. Feature id auto-allocated as `003` (folders `001/`, `002/` already exist). Times are illustrative.

**Step 0 — User invokes:**
```
/orchestra add password reset to login
```
`metrics-collector` logs `prompt.submitted` with `matched_orchestra: true`. `TeamCreate` instantiates the 8-agent team.

**Step 1 — Confidence classification (§8.11.1).**
- `intent_word_count = 6` → +HIGH
- `project_mode = greenfield` → +HIGH
- `files_touched_estimate = 8` (login route, auth service, mailer, two test files, two migrations, one component) → MEDIUM
- `intent_classification_match_score`: matches §9.5 Feature row → +HIGH

Net tier: **MEDIUM**. `@lead` selects Pattern B (one revision dialogue).

**Step 2 — Spec phase, Pattern B (§9.4).**
`@product` drafts `pipeline/003/PRD-003.md` and `pipeline/003/FRS-003.md`. `hash-stamper` runs on save: stamps `S-CONTEXT-001`, `S-USERSTORIES-001`, `S-ACCEPTANCE-001` hashes; references upstream `PRD-001` (this doc) → `S-INVARIANTS-001` with `hash-at-write: sha256:<...>`.

`@lead` reads, writes critique inline as `revision: 2`. `@product` revises FRS-003 (acceptance criterion C-3 reworded). Single revision; no circuit breaker.

**Step 3 — Architecture (§D-22).**
`@lead` decides feature *touches* `architecture/SAD.md` §S-CONTAINER-001 (adds mail container). Updates SAD; new section hash propagates. Drafts `pipeline/003/TDD-003.md` cloning SAD §S-CONTAINER-001 and §S-CONTEXT-001 into TDD body with hash-at-clone (D-23).

Drafts `pipeline/003/API-003.openapi.yaml` with `POST /auth/reset-request` and `POST /auth/reset-confirm`.

**Step 4 — Contract (§8.2).**
`@lead` drafts `pipeline/003/CONTRACT-003.md` with weighted criteria (sum=100):
- 30 — `http_probe POST /auth/reset-request → 202`
- 30 — `http_probe POST /auth/reset-confirm with valid token → 200`
- 20 — `db_state` confirms reset token is single-use (post-confirm row deleted)
- 20 — token expiry honored (60-min cap)

`@evaluator` validates probes, co-signs (`signed: true`). No revision needed.

**Step 5 — Implementation.**
`@backend` writes `IMPL-BE-003.md` (informational) + actual code (auth service, migrations, mailer integration). Cannot run tests (no Bash). Refines `API-003.openapi.yaml` against impl reality; `hash-stamper` re-stamps `S-API-001`.

`@frontend` writes `IMPL-FE-003.md` + actual code (reset-request form, confirm-token landing page) with all 4 states (loading/empty/error/success).

`@test` writes `pipeline/003/TEST-003.md` plan + adversarial fuzz (expired token, used token, malformed JWT, replay).

**Step 6 — Verdict.**
`@evaluator` runs probes via `orchestra-probe` MCP. Result: 4/4 criteria PASS. Writes `TEST-003.md` verdict block (`verdict: PASS`, weighted score 100/100). `@evaluator` cannot patch any failing test by design — if a probe failed, returns to `@backend` with diagnostic.

**Step 7 — Review.**
`@reviewer` reads diff + `IMPL-BE-003`/`IMPL-FE-003`. Writes `pipeline/003/CODE-REVIEW-003.md`. Verdict: `APPROVED` with 2 minor suggestions (informational). No REQUEST_CHANGES; no second loop.

**Step 8 — Ship.**
`@ship` writes `pipeline/003/DOC-003.md` (user-facing) + appends `## [1.1.0]` to `CHANGELOG.md`. `RUNBOOK-v1.1.0.md` is **skipped** (no topology change — no new container at runtime, just new endpoints). `RELEASE-v1.1.0.md` written with reference to `pipeline/003/`.

**Step 9 — Pre-release validation.**
`validate-drift.js` walks `pipeline/003/` + `architecture/SAD.md`. Every `hash-at-write` resolves; one `inferred:` warning on a SAD subsection (`severity: warn`); zero `confirmed:` drift. `DRIFT-REPORT.md` clean for release. `metrics-collector` logs `session.stopped` with token totals.

**Resulting `pipeline/003/` tree:**
```
pipeline/003/
├── PRD-003.md           (revision: 2)
├── FRS-003.md           (revision: 2)
├── API-003.openapi.yaml
├── TDD-003.md           (cloned SAD sections inline)
├── CODE-DESIGN-BE-003.md
├── CODE-DESIGN-FE-003.md
├── IMPL-BE-003.md
├── IMPL-FE-003.md
├── TEST-003.md          (plan + verdict: PASS)
├── CONTRACT-003.md      (signed: true)
├── CODE-REVIEW-003.md   (verdict: APPROVED)
├── DOC-003.md
└── PLAN-003.md
```

This run touches 8 of the 18 artifact types; absences (`DEADLOCK`, `ESCALATE-ARCH`, `RUNBOOK`, `ANNOUNCEMENT`) are correct for a clean MEDIUM-confidence feature on a greenfield project. Brownfield + LOW-confidence runs would trigger Pattern C (wave team in step 2) and likely produce more `inferred:` warnings in step 9.
